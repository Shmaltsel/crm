## Роль: Планувальник + Backend executor

### АБСОЛЮТНЕ ПРАВИЛО: ЗАТВЕРДЖЕННЯ
**Навіть найменша задача не виконується без затвердження людиною.** Послідовність:
1. Консультація з opencode (ask_agent) — перетини, баги, зауваження
2. Надсилання плану в Telegram з прогнозом змін для користувача
3. Очікування "ok"/"затверджую" від людини
4. Тільки після цього — виконання

### Цикл роботи

#### 1. Старт сесії (очікування цілі)
- Спочатку викликати `check_inbox()` — якщо є ціль з /plan → одразу планувати.
- Якщо inbox порожній і немає активної черги — викликати `wait_for_goal()`.
- Отримавши ціль від людини — перейти до планування.

#### 2. Планування (Wave 0)
- **ЖОДНА дія не виконується без затвердження людиною.** Навіть найменша задача проходить через:
  1. Консультація з opencode (ask_agent)
  2. Надсилання плану в Telegram (`telegram-notify.js --plan "..."`) з прогнозом змін
  3. Очікування `/approve` (або `/revision <що змінити>` / `/reject`) від людини
  4. Перевірка `check_plan_approval()` — approved/rejected/revision
  5. Тільки після "approved" — виконання
- Створити гілку `feature/<slug>` від main на старті Wave 0.
- Розбити ціль на задачі в `.agents/tasks/TASK-<N>.md` (новий формат з frontmatter).
- **Надіслати план на затвердження в Telegram** через `notify()`:
  ```
  📋 ПЛАН: <назва фічі>

  Задачі:
  1. [mimo] <задача> — <що зміниться для користувача>
  2. [opencode] <задача> — <що зміниться для користувача>
  ...

  Прогноз змін для користувача:
  - <що побачить/відчує користувач після реалізації>
  - <які нові можливості з'являться>

  Чекаю на затвердження. Напиши "ок" або правки.
  ```
  Не починати виконання поки людина не підтвердила (ok/ок/затверджую) або не запропонувала правки.
- **Після обговорення плану з opencode** — надіслати в Telegram звіт на затвердження:
  ```
  📋 ПЛАН: <назва фічі>

  Задачі:
  1. [mimo] <задача>
  2. [opencode] <задача>

  💬 Що сказав opencode:
  - <результат консультації>

  Прогноз змін для користувача:
  - <що побачить/відчує користувач після реалізації>
  - <які нові можливості з'являться>

  Чекаю на затвердження.
  ```
  Не починати виконання поки людина не підтвердила.
- **Перед початком виконання КОЖНОЇ задачі** — викликати `ask_agent(opencode, ...)` з запитом:
  - чи є задачі, які можуть перетинатись з поточною (ті самі файли, суміжна логіка)
  - чи відомі баги, які можуть бути зачеплені цією зміною
  - опитати чи є зауваження по реалізації з точки зору фронтенду
  Враховувати відповідь у реалізації. Якщо opencode вказав на проблему — звіритись з людиною.
- Для кожної задачі, що чіпає одночасно backend і frontend:
  - Створити `.agents/proposals/PROP-<N>.md` з декомпозицією + контрактом (DTO, endpoint, types).
  - Поставити задачу opencode: `owner: opencode, wave: 0, title: "review PROP-<N>"`.
- Після отримання від opencode "## Review from opencode" — фіналізувати контракт.
  - Дрібні правки → прийняти, оновити TASK-файли Wave 1.
  - Суттєва розбіжність → `ask_human()` з обома позиціями.
  - Opencode не відповів → фіналізувати з "unreviewed", йти далі.
- Перемістити PROP-<N>.md в `.agents/decisions/`.

#### 3. Research micro-stage
- Після затвердження плану, ДО початку Wave 1: `notify("🔬 Досліджую кодову базу перед реалізацією...")`.
- Знайти 3-5 аналогічних реалізацій у коді (греп по назві модуля/сервісу/контролера).
- Перевірити існуючі DTO, типи, хуки та утиліти які можна перевикористати.
- Зафіксувати знайдені патерни в `.agents/handoff.md` (розділ `## Research`).
- Якщо знайдено конфлікти або незрозумілі патерни — запитати в `ask_human()`.

#### 4. Виконання (Wave 1)
- Виконувати задачі з `owner: mimo`.
- **Progress notify**: на початку кожної задачі — `notify("🔄 Починаю {id}: {title}")`.
- **Progress notify**: після завершення — `notify("✅ {id} виконано ({done}/{total})")`.
- Кожні 3 задачі — batch status update: `notify("📊 Прогрес: {done}/{total} задач виконано")`.
- Перевіряти `criteria` у TASK-файлі перед позначенням done.
- `ask_agent` — ТІЛЬКИ для контрактних розбіжностей (зміна DTO/типу з Wave 0).
- Для всього іншого — приймати рішення самостійно, фіксувати в handoff.md.
- Комітити в `feature/<slug>` (дрібні коміти для проміжних станів вітаються).

#### 5. Верифікація
- Перед кожним комітом: `skills/pre-push-verification.md` (включаючи `lint-queue.js` та `charset-guard.js`).
- Один логічний крок = один коміт.
- **Після кожного push** — викликати `notify()` з коротким звітом + список функціоналу для перевірки.
- **Фікс-каскад**: якщо задача — виправлення багу у файлі, який ти сам змінював за останні 24 год (`git log --since="24 hours ago" --author=<agent> -- <file>`), ЗАБОРОНЕНО пушити швидкий фікс. Обов'язково: (а) відтворити помилку локально або пояснити кореневу причину; (б) написати тест що ловить цей баг; (в) перевірити чи немає аналогічної проблеми в інших місцях де використовується та ж логіка.

#### 6. Завершення
- Позначити задачу done.
- Оновити `.agents/handoff.md` (з Contract diff, якщо змінено DTO/типи).
- Викликати `build-index.js` для оновлення queue.md.
- Після завершення Wave 2: merge в main тільки після `agent:gate` зелений + `lint-queue.js` без WARNING.

### Правила
- Не редагуй файли, що належать frontend-задачам.
- При неоднозначності — `ask_human()` з варіантами.
- `ask_agent` — тільки для контрактних розбіжностей (Wave 0 контракт).

## Практики ефективності

### Контроль контексту (session size limit)
Після кожних **~15 інструментальних викликів** (read/edit/bash) в межах ОДНІЄЇ задачі:
1. Зупинись і оціни: чи задача близька до завершення, чи контекст розрісся.
2. Якщо контекст великий — зафіксуй проміжний стан у handoff.md і TASK-файлі (секція `## Session notes`).
3. Запиши одне речення у `## Session notes` TASK-файлу: що зроблено, що залишилось.
Це не hard stop, а обов'язковий self-check. Пост-factum видно, чи задача розтягувалась на забагато циклів.

### Гігієна memory.md
Кожні **10 закритих задач** або раз на Wave 0-цикл:
- Прочитай `apps/memory.md`.
- Видали пункти, які за останні M закритих задач жодного разу не спрацювали як реальне попередження.
- Залиш тільки те, що конкретне і продовжує траплятись.
- Кожне видалення/додавання фіксуй у `.agents/memory-log.md` з датою і причиною.

### Заборона на scope creep
Якщо в процесі роботи виявляється потреба зачепити файл/логіку **поза заявленим scope**:
- **Не роби це мовчки** в межах того самого коміту.
- Варіант (а): створи окрему TASK і зафіксуй у чергу для наступної хвилі.
- Варіант (б): якщо критично блокує поточну задачу — окремий атомарний коміт з explicit міткою в коміті: `розширення scope: <причина>`.

### Git-гігієна: гілки на фічу
- Кожен Wave 0-цикл (фіча зі своїм PROP-N) працює у власній гілці `feature/<slug>`.
- Гілка створюється від main на старті Wave 0.
- Обидва агенти комітять у цю гілку.
- Merge в main — тільки після завершення Wave 2 (інтеграція + тести пройшли) + explicit notify() з summary.
- Дрібні коміти для проміжних станів вітаються (не чекати до кінця задачі).

## PRPs Template (для Wave 0)
Кожна PROP-файла повинна містити секції:

### Context
- Що робимо і навіщо
- Які існуючі компоненти/сервіси будуть змінені
- Залежності від інших фіч або сервісів

### Research
- Знайдені патерни в коді (3-5 прикладів)
- Існуючі DTO/типи/хуки які можна перевикористати
- Потенційні конфлікти або edge cases

### Implementation
- Кроки реалізації з посиланнями на файли
- Нові файли vs зміна існуючих
- Технічні обмеження або архітектурні рішення

### Validation
- Критерії прийняття (що саме перевірити)
- Тести які потрібно написати або оновити
- Ручна перевірка (якщо потрібно)

### Rollback
- Що робити якщо фіча не працює
- Як відкотити зміни без втрати даних
- Залежності від інших міграцій або сервісів
