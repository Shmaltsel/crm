# SKILL: Bug Fixing Protocol (Локалізація та виправлення помилок)

**Коли використовувати**: користувач повідомляє про баг, помилку в консолі, краш UI або неправильну поведінку бізнес-логіки.

## Кроки (сувора послідовність)

1. **Збір інсайтів (Diagnostics)**
   - НЕ писати код одразу. 
   - Знайти всі пов'язані файли (Backend Controller/Service + Frontend Hook/Component).
   - Якщо помилка на фронтенді — перевірити, чи правильно передаються типи і чи відповідає DTO бекенду.
   - Якщо помилка на бекенді — перевірити логіку Prisma-запитів та валідацію `class-validator`.

2. **Формулювання гіпотези**
   - Коротко озвучити користувачу: "Проблема в X, тому що Y. Планую змінити Z."
   - Дочекатися підтвердження (якщо помилка неоднозначна) або діяти одразу (якщо це очевидна опечатка/type error).

3. **Точкове виправлення (Surgical Fix)**
   - Змінювати МІНІМАЛЬНУ кількість рядків коду. 
   - Не робити попутного рефакторингу "щоб було красивіше", якщо про це не просили.
   - Якщо баг пов'язаний з `Decimal` або серіалізацією дат — перевірити весь ланцюг (БД -> API -> Axios -> UI).

4. **Запобігання регресії (Regression Prevention)**
   - Перевірити, чи не зламає цей фікс інші місця, де використовується змінена функція/компонент (використовувати глобальний пошук).
