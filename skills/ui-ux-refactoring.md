# SKILL: Редизайн та UI/UX рефакторинг (Tailwind + Framer Motion)

**Коли використовувати**: користувач просить оновити дизайн, переробити верстку під мобільні пристрої (mobile-first), додати анімації або переписати існуючі компоненти інтерфейсу.

## 🚨 ПРАВИЛА ВИКОНАННЯ (ATOMIC & VISUAL PROTOCOL)
Агент ЗОБОВ'ЯЗАНИЙ дотримуватися цих 4 правил при кожній роботі з UI:

1. **Атомарність змін (One Task Per Step):** - Якщо користувач просить кілька змін одночасно (наприклад, *"додай тултипи і поправ відступи в таблиці"*), АГЕНТ НЕ МАЄ ПРАВА робити це в один прохід. 
   - Агент повинен розбити це на кроки: Крок 1 (Тултипи) -> Коміт -> Крок 2 (Відступи) -> Коміт. Це гарантує, що друга вимога не зламає першу.

2. **Запит референсів (Inspiration Prompting):** - Якщо користувач просить створити новий UI-блок чи дашборд з нуля, і не надав макету або скріншоту, Агент повинен зупинитися і запитати: *"Чи маєте ви скріншот-референс або Figma-макет бажаного результату? Якщо ні, я використаю стандартні компоненти системи"*.
   - Якщо референс надано, він є абсолютним пріоритетом для відступів та компонування.

3. **Кодовий Visual QA (Mobile-First):**
   - Адаптивність не з'являється магічним чином. Агент зобов'язаний явно писати `md:`, `lg:`, `xl:` класи для кожного блоку.
   - Завжди перевіряти переповнення тексту на мобільному (`truncate`, `overflow-hidden`, `whitespace-nowrap`).
   - Після завершення UI-задачі Агент зобов'язаний написати користувачу: *"✅ Готово. Будь ласка, зробіть Visual QA — перевірте, як це виглядає на мобільному розмірі екрану, перш ніж ми підемо далі"*.

4. **Суворі обмеження (Don'ts):**
   - **ЗАБОРОНЕНО** використовувати хардкод кольори Tailwind (наприклад, `text-slate-500`, `bg-blue-600`). Використовувати ВИКЛЮЧНО семантичні змінні проєкту (`text-content-secondary`, `bg-surface`, `bg-brand`).
   - **ЗАБОРОНЕНО** писати складні градієнти, тіні чи ефекти скла (glassmorphism), якщо це прямо не вказано користувачем. Тримати дизайн чистим (Flat / Clean UI).

## Кроки імплементації
1. **Аналіз структури:** Уникати "нескінченного сувою". Використовувати вкладки або Bottom Sheets (`MoreSheet.tsx`) для мобільного екрану.
2. **Анімації:** Використовувати готові `slideXVariants` чи `SPRING.snappy` з `src/lib/motion.ts`. Для жестів завжди додавати `dragConstraints`.
3. **Empty States:** Ніколи не рендерити просто порожній екран. Завжди додавати стан "Немає даних" (Empty State) з позитивним закликом до дії.
